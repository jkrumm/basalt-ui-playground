## Overview

All four content types (Blog, Docs, Guides, Blocks) run through a shared `defineCollection` factory
in `src/lib/collection.ts`. This page explains the internals — useful when adding a new content type
or debugging why a page isn't showing up.

---

## defineCollection

```typescript
export function defineCollection<TSchema extends ZodTypeAny>({
  schema,     // Zod schema — validates each MDX file's frontmatter
  meta,       // import.meta.glob result for frontmatter-only imports (eager)
  modules,    // import.meta.glob result for full module imports (eager)
  slugify,    // path → slug string
  sort,       // optional comparator for getAll()
}) { ... }
```

The factory returns an object with `getAll`, `getBySlug`, `getComponent`, `getHeadings`,
`getSitemapEntries`, `getSearchDocuments`, and a `meta` map for key-existence checks.

## Remark/Rehype Pipeline

Before `defineCollection` can read frontmatter, the MDX compiler must emit it as a named
export. Three remark plugins run in order during Vite's MDX compilation step:

```
1. remark-frontmatter      → Parses YAML block into file.data.matter (not yet exported)
2. remark-reading-time     → Injects readingTime into file.data.matter (must run before step 3)
3. remark-mdx-frontmatter  → Exports file.data.matter as: export const frontmatter = {...}
```

Two rehype plugins run after the remark pass:

```
4. rehype-slug             → Adds id attributes to h2/h3 headings
5. rehype-export-headings  → Exports: export const headings = [{id, text, depth}, ...]
```

This means each compiled `.mdx` file exposes three named exports:

```typescript
// Auto-generated by the plugin pipeline — never written by hand
export const frontmatter = { title: "...", publishedAt: "...", readingTime: "2 min read" };
export const headings = [{ id: "overview", text: "Overview", depth: 2 }];
export default MDXContent; // the React component
```

`import.meta.glob` can import any of these selectively:

```typescript
// Frontmatter-only (lightweight — no React component compiled)
const meta = import.meta.glob("../content/blog/*.mdx", { eager: true, import: "frontmatter" });

// Full module (React component + frontmatter + headings)
const modules = import.meta.glob<ContentModule<BlogFrontmatter>>("../content/blog/*.mdx", {
  eager: true,
});
```

`{ eager: true }` ensures all files load synchronously at module init time — required because
SSR loaders have no `await` budget for dynamic imports.

---

<Admonition type="warning" title="String literal requirement">
  `import.meta.glob()` calls must use string literals — the glob pattern cannot be a variable or
  function argument. Vite resolves glob patterns at build time via static analysis. This is why each
  content type has its own `_meta` and `_modules` declarations at the top level of `content.ts`
  rather than inside a factory function.
</Admonition>

## Slugify

The slugify function strips the file path down to the slug used in the URL:

```typescript
// Blog: ../content/blog/my-post.mdx → my-post
slugify: (path) => path.replace("../content/blog/", "").replace(".mdx", "");

// Docs: ../content/docs/getting-started/installation.mdx → getting-started/installation
slugify: (path) => path.replace("../content/docs/", "").replace(".mdx", "");
```

The slug is derived entirely from the filesystem path. To change a slug, rename the file with
`git mv`. There is no separate `slug` frontmatter field — that would create two sources of truth.

## Frontmatter validation

`defineCollection` validates every file's frontmatter against its Zod schema at startup:

```typescript
function validate(path: string, fm: unknown): TSchema["_output"] {
  const result = schema.safeParse(fm);
  if (!result.success) {
    const details = result.error.issues
      .map((e) => `  ${e.path.join(".")}: ${e.message}`)
      .join("\n");
    throw new Error(`[content] Invalid frontmatter in ${path}:\n${details}`);
  }
  return result.data;
}
```

`safeParse` returns `{ success, data }` or `{ success: false, error }` — never throws on its own.
Zod applies `.optional()` defaults automatically. Invalid files throw at startup so broken
frontmatter is caught immediately during development rather than silently skipped at runtime.

## Search index

`getSearchDocuments` maps each validated entry to a `SearchDocument` shape for the Fuse.js
full-text index. The mapping is defined per-collection, not in the factory:

```typescript
// In content.ts:
export function getSearchIndex(): SearchDocument[] {
  const blog = blogCollection.getSearchDocuments((slug, fm) => ({
    type: "blog" as const,
    slug,
    title: fm.title,
    description: fm.description,
    tags: fm.tags,
  }));
  // ... guides, docs, blocks
  return [...blog, ...docs, ...guides, ...blocks];
}
```

The search index is built lazily on first use in the `SearchModal` component and memoized for
the session.

## Sitemap entries

```typescript
export function getBlogSitemapEntries(): SitemapEntry[] {
  return blogCollection.getSitemapEntries((slug, fm) => ({
    url: `/blog/${slug}`,
    lastmod: fm.updatedAt ?? fm.publishedAt,
  }));
}
```

`lastmod` comes from frontmatter — if you revise a post, update `updatedAt` so Google sees
the change and re-crawls the page. Docs and Blocks omit `lastmod` (no dates in their schemas).

---

## Draft and noindex filtering

`draft: true` files are excluded from `getAll()`, `getSearchDocuments()`, and
`getSitemapEntries()`. They are still accessible directly by slug (the route's loader will
resolve them) — this allows preview links to work before a post is published.

`noindex: true` is not handled by the collection layer — it is emitted as a
`<meta name="robots" content="noindex">` tag in the route's `head()` function. The page remains
in the collection and sitemap unless `draft: true` is also set.
