## Architecture

The CBBI Blueprint is a Bun monorepo with three packages:

```
  Browser
  TanStack Start · Blueprint v6 · Jotai · HyperDX RUM
    |  EdenTreaty (typed HTTP, same-origin)
  Vite dev proxy /api  (prod: Bun.serve reverse proxy)
    |
  Elysia API :7713
  BetterAuth · Drizzle v1 · OTEL -> HyperDX ClickStack
    |  postgres.js
  PostgreSQL — basalt_ui_playground schema
```

## Key Design Decisions

### Why Bun?

Single runtime for everything — API server, build tooling, package management, test runner.
No node/npm/pnpm split. Bun workspaces replace pnpm workspaces with less configuration.

### Why Elysia over Hono?

End-to-end type safety with EdenTreaty. Elysia's type system propagates route definitions
to the client — changes to an API response type are immediately visible in frontend code.
Plugin ecosystem (CORS, OpenAPI, OpenTelemetry) is first-class.

### Why TanStack Start?

SSR-first React meta-framework with file-based routing and server functions. Built on
Vite 8 (not Vinxi/Nitro). Provides `beforeLoad` for auth guards and data prefetching
without render-then-redirect patterns.

### Why Blueprint v6?

Data-dense UI components designed for complex dashboards. Intent system (primary, success,
warning, danger) maps naturally to financial data visualization. Pairs with Tailwind for
layout utilities while Blueprint handles component styling.

### Why Jotai over Zustand?

Atomic state model fits naturally with React's component tree. No boilerplate stores —
atoms compose. `atomWithStorage` provides cookie-backed persistence for theme preferences
with SSR support.

### Why Drizzle v1 Beta?

Forward-looking choice. v1 beta API is the future; starting with it avoids a migration
later. `pgSchema` isolation keeps tables in `basalt_ui_playground` namespace, preventing
conflicts with other applications sharing the same database.
