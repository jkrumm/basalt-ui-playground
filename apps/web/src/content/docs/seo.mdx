## Overview

This project uses a layered SEO strategy: TanStack Start's native `head()` API for per-route
meta tags, JSON-LD structured data for machine readability, and dedicated crawlable endpoints
for search engines and LLMs.

All SEO signals are **opt-in, explicit, and reviewed on each content change** â€” nothing is generated
from defaults.

---

## Per-route meta via `head()`

TanStack Start exposes a `head()` function on every route. It runs server-side and injects into
`<head>` via `<HeadContent />` in `__root.tsx`. No third-party SEO library is needed.

**Blog post** (`src/routes/_content/blog/$slug.tsx`):

```ts
head: ({ loaderData: ld }) => ({
  meta: [
    { title: `${fm.title} â€” CBBI Blueprint` },
    { name: 'description', content: fm.description },
    { property: 'og:title', content: fm.title },
    { property: 'og:description', content: fm.description },
    { property: 'og:type', content: 'article' },
    { property: 'og:url', content: url },
    { property: 'og:image', content: fm.image },    // conditional
    { name: 'article:published_time', content: fm.publishedAt },
    { name: 'article:modified_time', content: fm.updatedAt },  // conditional
    { name: 'article:tag', content: tag },          // one per tag
    { name: 'twitter:card', content: 'summary_large_image' },
    { name: 'twitter:title', content: fm.title },
    { name: 'twitter:description', content: fm.description },
    { name: 'twitter:image', content: fm.image },   // conditional
  ],
  links: [{ rel: 'canonical', href: url }],
  scripts: [{ type: 'application/ld+json', children: JSON.stringify(jsonLd) }],
}),
```

**Review checklist:**
- [ ] Title â‰¤ 60 characters, unique per page
- [ ] Description 150â€“160 characters
- [ ] `og:image` points to a real, accessible URL (1200Ă—630px recommended)
- [ ] `article:published_time` matches actual publication date
- [ ] Canonical URL is correct and absolute

## JSON-LD structured data

Two layers of structured data are injected:

**Root (`__root.tsx`)** â€” site-level identity, loaded on every page:
- `WebSite` â€” name, URL
- `Person` â€” author identity with `sameAs` links to GitHub and other profiles

`sameAs` is critical for LLMs to resolve the author identity across sources. Keep it updated
when new authoritative profiles are added.

**Blog post (`$slug.tsx`)** â€” per-post data:
- `BlogPosting` â€” headline, description, dates, author, image, keywords

**Review checklist:**
- [ ] `sameAs` in `Person` schema includes all current authoritative profiles
- [ ] `BlogPosting` `dateModified` is updated when posts are revised
- [ ] Validate with [Google Rich Results Test](https://search.google.com/test/rich-results) after structural changes
- [ ] Add `Article` â†’ `isPartOf` referencing `#website` if Google Search Console reports orphaned articles

## Sitemap

**Production:** served by `server.ts` via direct import from `src/lib/sitemap.ts` and `src/lib/content.ts`.
**Development:** served by the `sitemap-dev` Vite middleware in `vite.config.ts`.

The sitemap is generated dynamically on each request from the content collections.
Key property: `<lastmod>` is sourced from `updatedAt` (or `publishedAt` as fallback) in each
MDX file's frontmatter â€” Google uses this to decide crawl frequency.

**Review checklist:**
- [ ] Every published content page appears in `/sitemap.xml`
- [ ] `<lastmod>` reflects the actual last edit date (update `updatedAt` in frontmatter when revising)
- [ ] `/sitemap.xml` is referenced in `public/robots.txt`
- [ ] After adding a new content type, add its `get*SitemapEntries()` function in `src/lib/content.ts`
  and wire it into both the Vite middleware and the `server.ts` route

## AI discoverability (llms.txt)

`/llms.txt` is a Markdown-formatted plain text file that tells LLM crawlers what this site
contains and where the key pages are. Adopted by 800k+ sites as of 2025 (Anthropic, Cloudflare,
Stripe).

**Production:** served by `server.ts`.
**Development:** served by the `llms-txt-dev` Vite middleware in `vite.config.ts`.
**Source:** `buildLlmsTxt()` in `src/lib/sitemap.ts`.

**Review checklist:**
- [ ] Description accurately reflects the site's current purpose
- [ ] Key page list is up to date (add new sections when launched)
- [ ] Verify at `/llms.txt` after deploy

## robots.txt

Located at `public/robots.txt`. Explicitly allows all crawlers including major AI agents:
`GPTBot`, `Claude-Web`, `Anthropic-AI`, `PerplexityBot`.

**Review checklist:**
- [ ] Sitemap URL in `robots.txt` matches the actual deployed domain
- [ ] No content pages are accidentally disallowed

## Slug conventions

Blog slugs are **pure keyword slugs** â€” no date prefix in the URL. Dates belong in frontmatter
(`publishedAt`, `updatedAt`), not URLs.

```
âś“  /blog/tanstack-start-seo-guide
âś—  /blog/2026-01-15-tanstack-start-seo-guide
```

The slug is derived from the MDX filename: `src/content/blog/my-post.mdx` â†’ `/blog/my-post`.
Rename the file with `git mv` to change a slug â€” there is no separate slug field.

**If you must change a slug after indexing:** add a redirect from the old URL to the new one
in `server.ts` (301 permanent redirect).

## Adding a new content type

When adding a new section (e.g. `/tutorials`):

1. Create the collection in `src/lib/content.ts` with a frontmatter schema
2. Add a `get*SitemapEntries()` function that returns `SitemapEntry[]` with `lastmod`
3. Wire it into `collectSitemapEntries(...)` in both:
   - The `sitemap-dev` Vite plugin in `vite.config.ts`
   - The `/sitemap.xml` handler in `server.ts`
4. Add the section to `buildLlmsTxt()` in `src/lib/sitemap.ts`
5. Add the route to the `prerender.filter` in `vite.config.ts` if it should be SSG
