## Environment strategy

| File              | Committed | Purpose                                                   |
| ----------------- | --------- | ---------------------------------------------------------- |
| `apps/api/.env`   | Yes       | API defaults + local dev credentials (no prod secrets)    |
| `apps/web/.env`   | Yes       | Web defaults — `VITE_*` vars, port                        |
| `.env.example`    | Yes       | Full reference with placeholder values                    |

Bun auto-loads `.env` from each app's directory — no `dotenv` package needed.
Production injects variables via Doppler — no `.env` files at runtime.

---

## Local dev credentials

`apps/api/.env` ships with ready-to-use local values:

```bash
DATABASE_URL=postgresql://basalt_ui_playground:<password>@localhost:5432/<main-db>
BETTER_AUTH_SECRET=<local-dev-secret>
```

The `basalt_ui_playground` Postgres user and schema are provisioned by the VPS
`setup-postgres.sh` script. Adjust credentials if your local setup differs.

| `VITE_UMAMI_WEBSITE_ID`       | Web          | — (disabled if empty)      | No                           |

`VITE_*` variables are exposed to the client bundle. All others are server-only.

## Analytics (Umami)

Umami is injected in `__root.tsx` only when both vars are set:

```typescript
function umamiScript() {
  const url = import.meta.env.VITE_UMAMI_SCRIPT_URL;
  const id = import.meta.env.VITE_UMAMI_WEBSITE_ID;
  if (!url || !id) return [];
  return [{ src: url, async: true, "data-website-id": id, "data-auto-track": "false" }];
}
```

`data-auto-track="false"` disables Umami's default pageview tracking. Pageviews are
tracked manually via `RouteTracker` in `src/components/analytics/RouteTracker.tsx` —
this gives precise control and allows filtering internal navigation.

Custom events are tracked via the typed `track()` utility:

```typescript
import { track, EVENTS } from "~/lib/analytics.ts";

track(EVENTS.VIEW_TOGGLED, { component: "indicator-grid", view: "table" });
```

See `src/lib/analytics.ts` for the full event registry and `CLAUDE.md` (web app) for
the event naming convention.

## Observability (HyperDX)

For self-hosted ClickStack, set the API endpoint:

```bash
# apps/api/.env — backend OTEL exporter
OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318   # VPS dev.compose.yml maps 4318:4318

# apps/web/.env — browser SDK
VITE_HYPERDX_ENDPOINT=http://localhost:7707          # VPS dev.compose.yml maps 7707:8080
VITE_HYPERDX_API_KEY=dev                             # any non-empty string (self-hosted ignores key)
```

For Docker, set `OTEL_EXPORTER_OTLP_ENDPOINT=http://clickstack:4318` in the api service
environment and `VITE_HYPERDX_ENDPOINT=http://localhost:8080` for browser access.

## Theme cookie

The selected theme (`light` or `dark`) is persisted in a plain cookie so SSR can read
it before rendering — eliminating the flash-of-wrong-theme that localStorage-based
systems cause:

```typescript
export const getThemeFn = createServerFn({ method: "GET" }).handler(async () => {
  const theme = getCookie("theme");
  if (theme === "light" || theme === "dark") return theme;
  return "dark" as const;
});
```

## Production server

`apps/web/server.ts` is a Bun.serve HTTP server that:

1. Proxies `/api/*` to `API_INTERNAL_URL` (API container or localhost)
2. Serves hashed static files from `dist/client/assets/` with immutable cache headers
3. Falls back to the TanStack Start SSR handler for all other routes

Start after building:

```bash
bun run build && make start
```
