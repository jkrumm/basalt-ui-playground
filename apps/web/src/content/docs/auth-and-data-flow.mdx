## Architecture Overview

TanStack Start (port 3000) and the Hono API (port 3001) are two separate processes.
BetterAuth runs on the Hono side — not inside TanStack Start. This is the split-backend
pattern.

```
Browser
  ↕  HTTP (port 3000)
TanStack Start :3000     ← React SSR/SSG + createServerFn
  ↕  HTTP (credentials: include) / in-process (SSR)
Hono API :3001           ← BetterAuth handler + /api/user/* routes
  ↕  postgres.js
Postgres                 ← sessions + users + preferences
```

This separation keeps auth concerns in the backend (Hono + BetterAuth) and frontend
concerns in TanStack Start. The API is independently deployable and reusable by other clients.

The session cookie is set by Hono on `localhost:3001`. In local development,
`credentials: 'include'` sends the cross-origin cookie to all API requests. In production,
both services share a domain so same-origin cookie rules apply automatically.

## BetterAuth Backend Setup

BetterAuth is configured in `apps/api/src/auth.ts` and mounted on the Hono router.
The Drizzle adapter auto-generates the `user`, `session`, `account`, and `verification`
tables via `@better-auth/cli generate`.

```typescript filename="apps/api/src/auth.ts"
export const auth = betterAuth({
  baseURL: env.BETTER_AUTH_URL,
  secret: env.BETTER_AUTH_SECRET,
  database: drizzleAdapter(db, { provider: "pg", schema: authSchema }),
  emailAndPassword: { enabled: true },
});
```

Mount the handler so all `POST/GET /api/auth/**` requests are handled by BetterAuth:

```typescript filename="apps/api/src/app.ts"
app.on(["POST", "GET"], "/api/auth/**", (c) => auth.handler(c.req.raw));
```

The auth middleware validates sessions for all `/api/user/*` routes. It caches the full
user object on the Hono context so handlers don't need a second DB lookup:

```typescript filename="apps/api/src/app.ts"
type Variables = { userId: string; user: BetterAuthUser };

const authMiddleware = createMiddleware<{ Variables: Variables }>(async (c, next) => {
  const session = await auth.api.getSession({ headers: c.req.raw.headers });
  if (!session) return c.json({ error: "Unauthorized" }, 401);
  c.set("userId", session.user.id);
  c.set("user", session.user as BetterAuthUser);
  await next();
});
```

## Protected Endpoints

Custom authenticated endpoints sit behind `authMiddleware`. They are registered on the
typed `userRoutes` sub-app so `hono/client` can infer their request/response types.

```typescript filename="apps/api/src/app.ts"
// GET /api/user/me — returns the authenticated user's profile
const getMeRoute = createRoute({
  method: "get",
  path: "/api/user/me",
  responses: {
    200: jsonContent(UserSchema, "Authenticated user profile"),
    401: jsonContent(z.object({ error: z.string() }), "Unauthorized"),
  },
});

const userRoutes = new OpenAPIHono<{ Variables: Variables }>().openapi(getMeRoute, (c) => {
  const u = c.get("user"); // no DB query — cached by middleware
  return c.json(
    {
      id: u.id,
      name: u.name,
      email: u.email,
      emailVerified: u.emailVerified,
      image: u.image ?? null,
      createdAt: u.createdAt.toISOString(),
    },
    200,
  );
});
// ...preferences routes
```

The full OpenAPI spec including these routes is available at `/doc` (JSON) and `/scalar`
(interactive UI) — no manual documentation required.

## Protected Routes (Frontend)

TanStack Start uses a pathless layout route (`_protected.tsx`) to guard all auth-required
pages. Any route nested under `_protected/` inherits the guard automatically.

```typescript filename="apps/web/src/routes/_protected.tsx"
export const Route = createFileRoute("/_protected")({
  beforeLoad: async ({ location }) => {
    const session = await getSessionFn();
    if (!session) throw redirect({ to: "/sign-in", search: { redirect: location.pathname } });
    return {
      user: session.user,
      // Serialize as ISO string — Date objects don't survive router hydration reliably
      sessionExpiresAt: session.session.expiresAt.toISOString(),
    };
  },
  component: () => <Outlet />,
});
```

`getSessionFn` is a `createServerFn` that forwards the request cookie to Hono for
session validation. The context is available to all child routes via `Route.useRouteContext()`.

## SSR Route Context Pattern

The `beforeLoad` return value is the cleanest way to pass SSR-loaded auth data to
protected routes — no extra fetch, no query waterfall, available on first render.

```typescript filename="apps/web/src/routes/_protected/account.tsx"
export const Route = createFileRoute("/_protected/account")({
  component: AccountPage,
});

function AccountPage() {
  // Loaded server-side in _protected.tsx — zero additional requests
  const { user, sessionExpiresAt } = Route.useRouteContext();

  return (
    <Card>
      <H4>{user.name}</H4>
      <p>{user.email}</p>
      <Tag intent={user.emailVerified ? "success" : "warning"}>
        {user.emailVerified ? "Verified" : "Not verified"}
      </Tag>
      <p>Session expires: {new Date(sessionExpiresAt).toLocaleString()}</p>
    </Card>
  );
}
```

**When to use route context vs. a query:**

| Data                          | Pattern                        | Reason                                          |
| ----------------------------- | ------------------------------ | ----------------------------------------------- |
| Auth identity (user, session) | Route context                  | Already fetched by `beforeLoad`; no waterfall   |
| User preferences              | `useSuspenseQuery`             | Changes independently; needs cache invalidation |
| Any data fetched by a loader  | Route context via `loaderData` | Loader result is the canonical pattern          |

## Frontend Auth Client

`createAuthClient()` from `better-auth/react` provides:

- `authClient.useSession()` — reactive session state (React Query-like)
- `authClient.signIn.email()` — sign in with credentials
- `authClient.signUp.email()` — create account
- `authClient.signOut()` — clears session cookie + updates reactive state

```typescript filename="apps/web/src/lib/auth-client.ts"
export const authClient = createAuthClient();
// No baseURL — same origin via Vite proxy in dev, Node server in prod
```

After sign-in, `router.invalidate()` forces `_protected` `beforeLoad` to re-run with the
new cookie, so protected routes become accessible immediately:

```typescript
const { error } = await authClient.signIn.email(value);
if (!error) {
  await router.invalidate();
  window.location.assign(redirect);
}
```

After sign-out, `router.invalidate()` ensures stale protected-route context is discarded:

```typescript
await authClient.signOut();
await router.invalidate();
navigate({ to: "/" });
```

## Server-Side Session Function

`getSessionFn` is a `createServerFn` — it always runs on the server, even when called
from client-side navigation. This ensures session validation always hits Hono, not the
browser's local state.

```typescript filename="apps/web/src/lib/auth.functions.ts"
export const getSessionFn = createServerFn({ method: "GET" }).handler(async () => {
  const headers = getRequestHeaders();
  const session = await auth.api.getSession({ headers });
  return session;
});
```

The request headers (including the session cookie) are captured via `getRequestHeaders()`
from `@tanstack/react-start/server` and forwarded to BetterAuth.

## hono/client — Isomorphic API Client

`hono/client` provides a fully-typed RPC client inferred directly from the Hono `AppType`.
No code generation required — TypeScript inference does all the work.

```typescript filename="apps/web/src/lib/api.ts"
export const getApi = createIsomorphicFn()
  .server(async () => {
    const { app } = await import("@cbbi/api");
    const reqHeaders = getRequestHeaders();
    return hc<AppType>("http://localhost", {
      fetch: (input, init) => {
        const merged = new Headers(reqHeaders);
        new Headers(init?.headers).forEach((v, k) => merged.set(k, v));
        return app.fetch(new Request(input as string, { ...init, headers: merged }));
      },
    });
  })
  .client(() =>
    hc<AppType>(import.meta.env.VITE_API_URL ?? "", {
      init: { credentials: "include" },
    }),
  );
```

`createIsomorphicFn` returns different implementations depending on where it runs:

| Context              | Implementation                              | Latency     |
| -------------------- | ------------------------------------------- | ----------- |
| **Server (SSR/SSG)** | Direct in-process Hono call via `app.fetch` | ~0ms        |
| **Client (browser)** | `fetch` to Hono API at `VITE_API_URL`       | network RTT |

The server branch merges incoming request headers so the auth cookie flows through
to BetterAuth even during in-process calls. Caller-set headers (e.g. `Content-Type`)
take precedence via the `merged.set(k, v)` override.

## Query Factory Pattern

All server state is defined as `queryOptions()` factory functions in `src/queries/`. Never
call `useQuery` with an inline `queryFn` — that duplicates the query key and prevents
cache sharing between components.

```typescript filename="apps/web/src/queries/preferences.queries.ts"
export const userPreferencesQuery = () =>
  queryOptions({
    queryKey: ["user", "preferences"],
    queryFn: async () => {
      const { data, error } = await getApi().user.preferences.get();
      if (error) throw error.value;
      return data!;
    },
  });
```

**Why factory functions?** They enable:

- Loader preloading: `loader: () => queryClient.ensureQueryData(userPreferencesQuery())`
- Consistent `queryKey` reuse across `invalidateQueries` and `useSuspenseQuery`
- Single place to update error handling, caching config, or the query key

## Zod End-to-End

Zod schemas defined in `packages/schemas/` flow through every layer without duplication:

```
packages/schemas/src/user.ts               → UserSchema (id, name, email, emailVerified, image, createdAt)
packages/schemas/src/user-preferences.ts  → UserPreferencesSchema + PatchUserPreferencesSchema
  ↓ imported by
apps/api/src/app.ts                         ← Hono route validation (request body + OpenAPI spec)
  ↓ via hono/client type inference
apps/web/src/lib/api.ts                     ← getApi() return type
  ↓ consumed in
apps/web/src/queries/preferences.queries.ts ← queryFn return type
  ↓ used in
apps/web/src/routes/_protected/settings.tsx ← useAppForm + zodFormValidator
```

The same schema validates the Hono API request (via `@hono/zod-openapi`), auto-generates the
OpenAPI spec at `/doc`, types the hono/client response, and drives TanStack Form validation
on the client. A breaking schema change surfaces at every layer simultaneously — TypeScript
catches it before runtime.
