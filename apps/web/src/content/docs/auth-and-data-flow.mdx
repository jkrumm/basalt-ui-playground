## Architecture Overview

The web app (port 7712) and API (port 7713) are separate processes. BetterAuth runs
on the Elysia side. The web app communicates with the API via EdenTreaty (typed HTTP)
and TanStack Start server functions.

```
Browser
  |  HTTP :7712
TanStack Start :7712    <-- React SSR + createServerFn
  |  HTTP (same-origin via proxy) or internal URL (SSR)
Elysia API :7713        <-- BetterAuth + /api/user/* routes
  |  postgres.js
PostgreSQL (basalt_ui_playground schema)
```

In development, Vite's `server.proxy` forwards `/api/*` from `:7712` to `:7713`.
In production, `apps/web/server.ts` (Bun.serve) proxies `/api/*` to
`API_INTERNAL_URL=http://api:7713` (Docker service name) or `http://localhost:7713`.

## Auth Session in SSR

During server-side rendering the browser's session cookie is in the incoming request
headers. A server function forwards it to the API to resolve the session:

```typescript
// apps/web/src/lib/auth.functions.ts
export const getSessionFn = createServerFn({ method: "GET" }).handler(async () => {
  const cookie = getRequestHeader("cookie") ?? "";
  const apiUrl = process.env["API_INTERNAL_URL"] ?? "http://localhost:7713";

  // Inject W3C traceparent so SSR -> API spans are linked in HyperDX
  const traceHeaders: Record<string, string> = {};
  propagation.inject(context.active(), traceHeaders);

  const response = await fetch(`${apiUrl}/api/auth/get-session`, {
    headers: { cookie, ...traceHeaders },
  });
  if (!response.ok) return null;
  return response.json();
});
```

Protected routes call `getSessionFn()` in `beforeLoad` — no render-then-redirect flash:

```typescript
// apps/web/src/routes/_protected.tsx
export const Route = createFileRoute("/_protected")({
  beforeLoad: async ({ location }) => {
    const session = await getSessionFn();
    if (!session) {
      throw redirect({ to: "/sign-in", search: { redirect: location.href } });
    }
    return { user: session.user };
  },
});
```

Child routes under `_protected/` inherit `user` from route context via `Route.useRouteContext()`.

## Client-Side Auth Actions

Sign-in, sign-out, and registration use the BetterAuth React client directly:

```typescript
// apps/web/src/lib/auth-client.ts
export const authClient = createAuthClient();
// No baseURL — same-origin via Vite proxy in dev, Bun.serve proxy in prod
```

```typescript
// Sign in
await authClient.signIn.email({ email, password });

// Sign out
await authClient.signOut();
await router.navigate({ to: "/" });
```

After sign-in, the session cookie is set by the API. Navigating causes TanStack Router
to re-run `beforeLoad` on the protected layout, picking up the new session.

## EdenTreaty API Client

EdenTreaty provides fully-typed API access with no codegen. The `App` type is exported
from `apps/api/src/app.ts` and consumed by the web workspace via `workspace:*`:

```typescript
// apps/web/src/lib/api.ts
import { treaty } from "@elysiajs/eden";
import type { App } from "@basalt-ui-playground/api";

function getBaseUrl(): string {
  // SSR: use internal URL to skip the network round-trip
  if (typeof window === "undefined") {
    return process.env["API_INTERNAL_URL"] ?? "http://localhost:7713";
  }
  // Client: same-origin (proxied in dev, Bun.serve in prod)
  return "";
}

export const api = treaty<App>(getBaseUrl(), {
  fetch: { credentials: "include" },
});
```

Route handler return types flow through automatically:

```typescript
const { data, error } = await api.api.user.me.get();
// data: { id: string; name: string; email: string; ... } | undefined
```

## Data Fetching Patterns

**Route loaders (SSR):** Use `createServerFn` for data needed at render time:

```typescript
const getCBBIData = createServerFn({ method: "GET" }).handler(async () => {
  const res = await fetch("https://colintalkscrypto.com/cbbi/data/latest.json");
  return res.json();
});

export const Route = createFileRoute("/")({
  loader: () => getCBBIData(),
  component: Dashboard,
});
```

**Authenticated requests:** Use EdenTreaty inside a server function to call the API
with the forwarded session cookie.

**Client-side mutations:** Use EdenTreaty directly in event handlers. For optimistic
updates, combine with TanStack Query's `useMutation`.

## Session Persistence

Session cookies are managed by BetterAuth — `HttpOnly`, set by the API on sign-in.
The web app never touches them directly; server functions forward them transparently.

Theme preference uses a plain (non-HttpOnly) cookie so SSR can read it for correct
first-render dark/light mode without a round-trip:

```typescript
export const getThemeFn = createServerFn({ method: "GET" }).handler(async () => {
  const theme = getCookie("theme");
  if (theme === "light" || theme === "dark") return theme;
  return "dark" as const;
});
```

## Zod End-to-End

Shared schemas in `packages/schemas/` flow through every layer without duplication:

```
packages/schemas/src/user.ts               -> UserSchema
packages/schemas/src/user-preferences.ts  -> UserPreferencesSchema
  imported by
apps/api/src/routes/user.ts                <- Elysia route body validation
  exported via App type
apps/web/src/lib/api.ts                    <- EdenTreaty inference
  consumed in
apps/web/src/routes/*.tsx                  <- typed component props
```

A breaking schema change surfaces across all layers simultaneously — TypeScript
catches it at every boundary before runtime.
