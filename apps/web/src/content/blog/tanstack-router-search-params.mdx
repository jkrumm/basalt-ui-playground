## The problem with `useState` for filters

Most filtering UIs start as `useState`. The user picks a tag, the list filters, and everything works — until they try to share the URL, refresh the page, or hit the back button. The filter disappears. The state was never in the URL.

The fix is to move that state into the URL. TanStack Router makes this type-safe and composable via `validateSearch`.

<Admonition type="note" title="Project convention">
  This project uses TypeBox for search param validation — not Zod. The API is identical in shape but the import paths differ. See the CLAUDE.md entry on URL state.
</Admonition>

## Defining a search schema

```typescript filename="routes/_content/blog/index.tsx"
import { Type, type Static } from '@sinclair/typebox'
import { Value } from '@sinclair/typebox/value'

const BlogSearch = Type.Object({
  tag: Type.String({ default: '' }),
})
type BlogSearch = Static<typeof BlogSearch>

export const Route = createFileRoute('/_content/blog/')({
  validateSearch: (raw: Record<string, unknown>): BlogSearch => {
    const result = Value.Default(BlogSearch, { ...raw })
    return Value.Check(BlogSearch, result) ? (result as BlogSearch) : { tag: '' }
  },
})
```

`Value.Default` fills missing fields with their schema defaults. `Value.Check` validates the result — if anything is invalid, fall back to `{ tag: '' }` rather than throwing.

## Reading the param

```typescript filename="components/BlogTagFilter.tsx"
function BlogTagFilter() {
  const { tag } = Route.useSearch()

  return (
    <div>
      {TAGS.map(t => (
        <Link
          key={t}
          to="/blog"
          search={{ tag: tag === t ? '' : t }}  // toggle: active tag clears itself
        >
          {t}
        </Link>
      ))}
    </div>
  )
}
```

The `search` prop on `<Link>` is fully typed — TypeScript will error if you pass a key that isn't in `BlogSearch`.

## Multiple params

Extend the schema with a sort order:

```typescript
const BlogSearch = Type.Object({
  tag:  Type.String({ default: '' }),
  sort: Type.Union(
    [Type.Literal('newest'), Type.Literal('oldest')],
    { default: 'newest' }
  ),
})
```

TanStack Router serializes the entire search object into the URL automatically. `?tag=bitcoin&sort=oldest` works with no additional configuration.

<Admonition type="warning" title="loaderDeps for loader-dependent params">
  If your loader uses the search param (e.g., to fetch filtered data from an API), add `loaderDeps`:

  ```typescript
  loaderDeps: ({ search }) => ({ tag: search.tag }),
  loader: ({ deps }) => fetchPosts({ tag: deps.tag }),
  ```

  Without `loaderDeps`, the loader will not re-run when the search param changes.
</Admonition>

## When to use URL state vs `useState`

| Signal | URL state | `useState` |
|-|-|-|
| Can the user share this view? | yes → URL | — |
| Should the back button restore it? | yes → URL | — |
| Is it tied to the current render only? | — | yes → state |
| Is it ephemeral animation/hover? | — | yes → state |

The rule in this project: if the state is in the URL it's a pageview. No custom analytics event needed — Umami captures it automatically from the URL change.

## Navigating programmatically

```typescript
const navigate = useNavigate({ from: '/blog' })

// Clear filter
navigate({ search: prev => ({ ...prev, tag: '' }) })

// Set filter
navigate({ search: prev => ({ ...prev, tag: 'bitcoin' }) })
```

The `from` option scopes the navigation to the current route, giving you autocomplete and type inference on the `search` object.

## Conclusion

Search params are the right home for any UI state that changes the view the user sees. TanStack Router's `validateSearch` + TypeBox gives you type-safety, default values, and graceful fallbacks — all without a state management library.
